How to get your Foursquare data

There are two ways to export your Foursquare/Swarm checkin history. Either method produces JSON files that work with Check-Ins-FM.

Option 1: Privacy data request

The official way. No coding required, but it can take several days to receive your data.

  1. Go to Foursquare's data request page
  2. Follow the instructions to request your data export
  3. You'll receive an email with a download link when ready (up to 7 days)
  4. Extract the zip — upload the checkins*.json files

Option 2: API export via Pinback

Faster and immediate, but requires a Foursquare developer account.

  1. Visit github.com/lokesh/pinback
  2. Follow the setup instructions to create a Foursquare API app and get an OAuth token
  3. Run Pinback to export your checkins as JSON
  4. Upload the resulting file here

Note: Foursquare is sunsetting City Guide, so the privacy request process may change. The API method via Pinback is a reliable alternative.

Privacy

Your data stays in your browser

All processing runs client-side via Pyodide (Python in WebAssembly). Your Last.fm and Foursquare files are never uploaded to any server.

Offline geocoding

Most coordinates are resolved to city names using offline-geocode-city, loaded from esm.sh. This library runs entirely in your browser with embedded city data — no coordinates are sent externally. The library code is fetched once from the CDN on page load.

Nominatim (fallback)

For airports and edge cases the offline geocoder can't resolve, coordinates rounded to ~10 km are sent to OpenStreetMap's Nominatim service. See the OSM Foundation privacy policy.

Pre-built location cache

A geo_seed.json file is loaded to speed up geocoding. It contains only coordinate-to-city mappings (e.g. "40.1,-74.2" → "New York, US") at ~10 km grid resolution, derived from the developer's own checkin data. It contains no venue names, timestamps, or personally identifying information. When this cache already has a match, no Nominatim request is made.

Local storage

Geocoding results are cached in your browser's localStorage so subsequent runs don't repeat lookups. You can clear this anytime via your browser's developer tools.

No analytics or tracking

This site does not use cookies, analytics, or any third-party tracking scripts.

How auto-detect works

The algorithm

Auto-detect analyzes your Foursquare checkin history to figure out where you lived and when. It uses a 120-day rolling window that walks through your checkins day by day. At each month boundary, the city with the most checkins in that window (at least 45%) is marked as your "home" for that month.

Consecutive months with the same home city are grouped into periods. Short periods (under 4 months) are filtered out as likely trips.

When it works well

Auto-detect is most reliable when you check in regularly (at least a few times per month). It handles major life moves well — if you moved from one city to another and kept checking in, it will find the transition.

When it struggles

  • Sparse checkins: If your checkin frequency dropped significantly (e.g. from 50/month to 1-2/month), the algorithm can't reliably detect home changes. A single trip with a few checkins may appear as a "home" city.
  • Multiple cities: If you split time between cities without a clear dominant one, results may be unstable.
  • Geocoding issues: Some locations resolve to county or district names instead of city names (e.g. "Travis County" instead of "Austin"). These are filtered out, but edge cases may slip through.

Recommendation

If you've lived in multiple cities — especially after your checkin frequency dropped — manual entry is more accurate. Use the form above to enter your cities with approximate date ranges. You don't need exact dates; YYYY-MM is fine.
